# Library / SDK Heuristics

For packages designed to be consumed by other code: language libraries, SDKs for external services, framework plugins, NPM/PyPI/crates.io distributions.

## When this applies

Concrete signals (need â‰Ą 2):

- `package.json` has `main` / `exports` / `types` but no `start` / `serve` script.
- `pyproject.toml` has `project.scripts` that are CLI commands, OR no scripts at all (pure library).
- `Cargo.toml` has `lib` target.
- `go.mod` with no `cmd/` or `main.go` at top (or minimal one for examples).
- `README.md` is dominated by API examples / "how to install and use", not deploy instructions.
- No HTTP server bootstrap.
- Tests dominate test execution time over runtime.

## Typical top-level modules

Libraries tend to have **fewer** top-level modules than web apps â€” most code is in one or two cohesive units. A library with 10+ top-level modules is usually a framework, not a library.

| Typical module | One-line responsibility |
|---|---|
| `public-api` (or `api`) | The exported surface â€” what consumers depend on |
| `core` | The library's primary business logic |
| `transport` | If the library wraps a network protocol (HTTP, WebSocket, RPC) â€” abstraction over fetch/curl/grpc |
| `codec` (or `serialization`) | Encoding / decoding to wire format / file format / schema validation |
| `types` (or `models`) | Type / model definitions shared across core (in typed languages) |
| `resilience` (or `retry-circuit`) | Retries, backoff, circuit breakers, rate limiting (if applicable) |
| `auth-adapters` | If the library handles multiple auth methods (API key, OAuth, mTLS) â€” each adapter is sub-module |
| `errors` | Library-specific error types (often a shared module) |
| `internal` (or `private`) | Internal-only utilities â€” modules NOT exported as public API |
| `examples` | Usage examples shipped in repo (sometimes excluded from MTM if just demo code) |
| `tests` | Test infrastructure (if shared across test files) |

## Common boundary pitfalls

| Pitfall | Why it's wrong |
|---|---|
| Splitting `public-api` from `core` then treating them independently | Public API is the *facade* over core. Changes to `core` invariants ripple to `public-api` contracts. They are tightly coupled â€” but worth separating because public API has stability commitments core does not. |
| Treating `internal` as one big module | `internal` is just "not exported"; the contents may have multiple distinct responsibilities. Sub-split as needed. |
| Lumping `types` everywhere | In strongly-typed languages, types are often shared. But if types are domain-specific to a module, keep them in that module. A `types` top-level module is usually only justified when types are truly cross-cutting (e.g., a `Result<T,E>` lib-wide). |
| Forgetting `resilience` when it's separately implemented | Many libraries put retries + backoff in a dedicated module. Don't fold it into `transport` if the implementations are separable. |
| Treating examples as production code | Examples often violate library invariants (hardcoded credentials, no error handling) for clarity. Mark them `excluded` or as a clearly-labeled module. |
| Splitting by language binding (e.g., `python-binding`, `node-binding`) when bindings are autogenerated | Auto-generated bindings are inventory items, not top-level modules. The generator + source schema is the real module. |

## Naming conventions

- Use `public-api` for the explicit external surface; `core` for the implementation that public-api delegates to.
- Use `transport` for network plumbing; `codec` for byte-level serialization. Don't conflate them.
- `internal` is fine as a slug for the "not exported" bag; alternatively `private` if the language convention prefers (e.g., Python `_module`).

## Risk hotspots â€” almost always need `RISKS.md` entries

- Semver-breaking changes to the public API surface (any non-additive change to `public-api`).
- Backwards compatibility shims for old API versions.
- Serialization format changes (wire incompatibility with deployed consumers).
- Cryptographic operations in transport (TLS configs, signing).
- Default values that consumers might depend on without realizing.
- Side effects in import-time code (initialization order issues).
- Threading / async assumptions baked into core (changing them breaks downstream).
- License compatibility of any vendored / inlined upstream code.

## Notes specific to this domain

- **Public API stability is the central organizing principle**. The wiki for a library should make it trivial to answer: "if I change X, does it break consumers?". Every module's `Public Surface` section is especially important here.
- **Semver discipline** in the project should be reflected in module-level `Modification Guide`. Each module should classify which changes are semver-patch / minor / major.
- Consumer code is NOT in the repo. Inbound interactions for `public-api` mostly say "external consumers via npm/PyPI/crates" â€” record this as an external label per the interaction-map convention, not as a fake module.
- If the library has multiple separately-published packages in one repo (monorepo of libraries), each published package may be its own top-level module â€” but only if their release cycles are decoupled. Otherwise treat them as siblings under one top-level.
